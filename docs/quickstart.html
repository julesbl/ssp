SSP Quick Start Guide

Introduction

This guide is aimed at people with a good level of php knowlege and some idea of how to install applications on shared hosts.

The objective of these routines is to supply a secure login system to be wrapped round applications and websites so that developers do not need to write their own.

These libraries have been hardened against sql injection, javascript injection, cookie and session theft and cross site scripting. The best protection will occure if the login form and the session and random cookies all use ssl, this will take the purchase of a certificate from the likes of verizone and configuration of the hosting to use https://.

Requirements

These routines need at least php version 4.1, preferably 4.3 for improved security. If you are using PEAR the latest updates need to be installed.

Works with MySql and MS Sql Server, nearly working on Posgre Sql.

Libraries

The routines can use either one of two sets of data abstraction libraries.

AdoDb

The libraries are configured to use this by default and they are included in the distribution in /sspincludes/adodb/.

See adodb.sourceforge.net/ for full documentation.

Directories

Place the directory sspadmin in the browser visible part of your hosting so that the applications there can be executed using the browser.

Place the sspincludes directory, if possible, outside the browser viewable area.

Database

Using either the commend line or your favourite database administration tool execute the SSP_Database.sql file stored in the database directory to create the database.

Another way is to execute dbsetup.php in sspadmin/setup once you have got the include paths and the database dsn correct.

There is also a dbsetupMSSQL.php for Microsft sequel server installations.

Include Paths

Finding my absolute paths

To find where the domain directory is located can be tough, the easiest way I have found is to create a php file with an incorrect include and the error reply should give you the absolute path to the web viewable directory, the non viewable area is usually in the directory above this and is usually where the ftp base directory is located for your login.

There are exceptions to that such as pear where the non web visible area is in a totally different directory and is connected to the viewable area by a link, the paths required for these hostings will either be in the ftp package path at the top or findable in the hosting firms documentation.

Modify the includeheader.change.php

Rename to includeheader.php

There are two options for the include paths, the development server and the live server, if you only have a live server just change the second set of paths or make them both the same.

Set $SSP_RootPath to the domain directory eg. /user/www/domainName/
Set $SSP_IncludePath to eg. $SSP_RootPath . "pathtoinclude/sspincludes"

Modify config.change.php

Rename to config.php

Go through config.php modifying the configuration as needed.

Primary importance is:

  1. The database DSN, see http://pear.php.net/manual/en/package.database.db.intro-dsn.php for full details.
  2. The domain setups, use .domain.com or whatever, the period at the start will mean it works with www.domain.com or something.domain.com or domain.com.
  3. The error log file location, preferably outsie the browser viewable area.
  4. Session and random cookie names.
  5. The random string and encryption strings.

Once you have chosen all the majority of the set ups as you need upload to the sspinclude directory.

Rename userclasses.change.php

Rename to userclasses.php . This file is where the developer can override parts of the main classes.

Database Creation and Initialisation

Once you have got the include and config files correct point the browser at the correct file in the setup directory in sspadmin.

dbsetup.php - MySql
dbsetupMSSql.php - MS Sql Server
dbsetupPostgreSql.php - Postgre Sql (experimental)

The basic admin user will be created as explained in the emrgencyadmin create.

Emrgency Admin Creation

Point you browser at emergencyadmincreate.php in sspadmin/setup directory, this will creat an admin user with the following details:

  • email: admin@admin.com,
  • username: admin
  • password: password100

This will only work if there are no admin users defined in the system.

Finally

Navigate to index.php in top level, you should see information on the session, if you see a require errors check your paths, if you see database errors check your DSN.

Delete the setup directory from the server!!!

Usage

Simply put:

require(path_to_sspadmin/includeheader.php);
$session = new SSP_Protect("user");

at the top of any php file you wish to protect, the access can be blank, user or admin or any addtional levels you put in the config file.

See the examples directory and the manual for more help, the user admin section is also worth a look.
